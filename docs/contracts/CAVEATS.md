A caveat is not a tolerance escape hatch: it must name the affected quantity,
the current reason for divergence, an owning work item, and a concrete exit
criterion.

Both converge in the many-voxels-with-identical-ρ limit; on real data with per-voxel heterogeneity the AR(1) task betas differ by ~10-20% of typical effect magnitudes while per-voxel beta correlation stays > 0.99 and intercepts agree to < 1e-3 relative. Neither algorithm is "wrong" — they are different pooling strategies. Users comparing fmrimod and Nilearn t/F maps should expect this magnitude of difference even on identical inputs. Exit: file a separate work-request to add a noise-pool quantisation path to fmrimod for users who want exact Nilearn parity; until then, document the divergence as the closing criterion. |

## Maintenance Rule

Generated reports may introduce a new caveat only after this file is updated in
the same change. The test suite checks that every structured `caveat_id` in the
checked-in parity reports appears in this index.
Rows in this index must also have live owning work items unless they explicitly
name a closed replacement owner in the exit criterion.
